# Bulk Add Attendees via Excel — Design

**Date:** 2026-05-06
**Author:** Brainstormed with Claude (`brainstorming` skill)
**Status:** Approved (pending written-spec review)

---

## 1. Goal

Add a second attendee-onboarding path for organizers, alongside the existing single-invite flow:

> Organizer downloads an Excel template, fills it with one row per person (each tagged Attendee / Sponsor / Key Speaker / Key Organizer), uploads it. New attendees are created in the DB silently — no email is sent. Sponsors, Key Speakers, and Key Organizers get a star overlay on their profile picture and a colored category chip wherever they appear in the networking UI (directory, public profile, connections list, suggestions).

Existing flows (single-invite, self-registration, OTP login, attendee wizard, connections, vCard, smart-matching, analytics, exports, announcements) MUST stay unchanged in behavior. The only behavioral change to existing code is that attendee listings now also return a `role` field, and existing UI surfaces render a badge for non-default roles.

---

## 2. Decisions locked during brainstorming

| Decision | Choice |
|---|---|
| Role cardinality | One category per person (single enum field) |
| Email behavior on import | Silent — no emails sent. Imported people log in via the normal OTP flow when they want to. |
| Excel template scope | Full profile fields (Category + first/last/email/phone/designation/company/industry/business type/city + optional LinkedIn / Website / Photo URL) |
| Duplicate handling | Skip the row, report it in the response. Never overwrite existing attendees. |
| Badge style | Star overlay on the bottom-right of the avatar + colored chip beside the name (amber Sponsor, indigo Speaker, purple Organizer) |
| Implementation approach | **B** — add `AttendeeRole` enum column + bulk-import endpoint + shared `<RoleBadge>` component. Synchronous parse/insert; no queue. |

## 3. Architecture

### 3.1 Schema change

`apps/api/prisma/schema.prisma`:

```prisma
enum AttendeeRole {
  ATTENDEE
  SPONSOR
  KEY_SPEAKER
  KEY_ORGANIZER
}

model Attendee {
  // ... existing fields untouched ...
  role AttendeeRole @default(ATTENDEE) @map("role")

  @@index([eventId, role])
}
```

A new Prisma migration adds the column with default `ATTENDEE`. Every existing attendee retroactively gets the safe default — no behavioral change for legacy rows.

### 3.2 API surface

All new endpoints live in the existing `attendees` module (`apps/api/src/attendees/`).

| Method | Path | Auth | Purpose |
|---|---|---|---|
| `GET` | `/events/:eventId/attendees/import-template` | ORGANISER | Streams a generated `.xlsx` template. |
| `POST` | `/events/:eventId/attendees/import` | ORGANISER | Multipart upload of an `.xlsx`. Parses, validates, inserts. Returns `{ added, skipped, errors }`. |

Existing endpoints stay exactly as they are. The existing `GET /events/:eventId/attendees` listing's response gains one new field per attendee: `role: AttendeeRole`. No request shape change, no breaking change for any existing consumer.

### 3.3 Frontend surface

| File | Change |
|---|---|
| `apps/web/src/app/(organiser)/events/[eventId]/attendees/page.tsx` | Add a second "Add attendees" button beside "Invite attendee". Add a Role column to the table with a colored chip. |
| `apps/web/src/components/organiser/BulkImportAttendeesModal.tsx` (new) | 2-step modal: download template → upload → results. |
| `apps/web/src/components/profile/RoleBadge.tsx` (new) | Shared component: star overlay + colored chip. Used by directory, profile, connections, suggestions. |
| `apps/web/src/app/(attendee)/directory/page.tsx` | Slot `<RoleBadge>` next to the avatar/name. |
| `apps/web/src/app/(attendee)/profile/[attendeeId]/page.tsx` | Slot `<RoleBadge>` near the profile header. |
| `apps/web/src/app/(attendee)/connections/page.tsx` | Slot `<RoleBadge>` on each connection row. |
| `apps/web/src/app/(attendee)/suggestions/page.tsx` | Slot `<RoleBadge>` on each suggestion card. |

### 3.4 What stays untouched (preserving existing magic)

Single-invite flow (`POST /events/:eventId/invite-attendee`), OTP login, attendee wizard, registration, connection requests, vCard, smart-matching, analytics, exports, announcements — none of these are touched. Existing rows default to `ATTENDEE` and look identical to before.

## 4. Excel template

Generated by `GET /events/:eventId/attendees/import-template`. Single visible sheet `Attendees`. Bold + frozen header row. Sensible column widths. A second hidden sheet `_Instructions` carries usage notes.

| Column | Required | Validation | Notes |
|---|---|---|---|
| Category | yes | Excel data-validation dropdown: `Attendee`, `Sponsor`, `Key Speaker`, `Key Organizer` | Server re-validates on upload. |
| First Name | yes | non-empty, ≤80 chars | |
| Last Name | yes | non-empty, ≤80 chars | |
| Email | yes | RFC email, lowercased on import | Unique within the event. |
| Phone | yes | non-empty | Stored as-is (matches existing register flow). |
| Designation | yes | non-empty | |
| Company | yes | non-empty | |
| Industry | yes | non-empty | |
| Business Type | yes | non-empty | |
| City | yes | non-empty | |
| LinkedIn URL | optional | URL with `http`/`https` if present | |
| Website URL | optional | URL with `http`/`https` if present | |
| Photo URL | optional | URL with `http`/`https` if present | Stored directly into `profilePhotoUrl`. |

Three sample rows are pre-filled (one Sponsor, one Speaker, one Attendee) so the organizer sees what good data looks like.

Imported rows that pass validation get `consentGiven = true`, `consentedAt = now()`, `profileCompleted = true`, and `role` mapped from the Category cell. The organizer's act of importing is the consent record.

Limits: max 1000 data rows per upload, max file size 5 MB.

## 5. Import modal flow

Two-step wizard inside one modal — no page navigation.

**Step 1 — Download template.** Single primary CTA: "Download template.xlsx". Secondary "Skip → Upload" link for organizers who already have a filled file from a previous batch.

**Step 2 — Upload.** Drag-and-drop area + click-to-browse. Client-side reject for non-`.xlsx` files (saves a server round-trip). "Back" returns to Step 1.

## 6. Server-side data flow

One HTTP request, synchronous, all-or-some semantics. The whole batch is never aborted for a few bad rows — they're reported individually.

1. **Auth check** — verify the organiser owns `eventId` (same pattern as `inviteAttendee`).
2. **Multer parse** — 5 MB file-size cap, mimetype guard for `.xlsx`.
3. **Workbook parse** — `await new ExcelJS.Workbook().xlsx.load(file.buffer)`. Read the `Attendees` worksheet. Missing worksheet → `400 INVALID_TEMPLATE`.
4. **Header validation** — row 1 must match the expected headers exactly. Renamed/missing/extra columns → `400 INVALID_TEMPLATE` with the offending column names. This is the only "abort the whole batch" case.
5. **Row count guard** — > 1000 data rows → `413 TOO_MANY_ROWS`.
6. **Pre-fetch existing emails** — single query: `SELECT email FROM attendees WHERE eventId = $1`. Stored lowercased into a `Set<string>` for O(1) duplicate checks.
7. **Per-row processing** — iterate rows. For each:
   - Trim + lowercase email.
   - Validate: required fields present, category in allowed set, email matches RFC pattern, optional URLs are `http`/`https` if present.
   - Validation failure → push `{ row, reason, value? }` into `errors[]`, continue.
   - Email in existing-set → push `{ row, email, reason: 'ALREADY_REGISTERED' }` into `skipped[]`, continue.
   - Email seen earlier in this same upload → push `{ row, email, reason: 'DUPLICATE_IN_FILE' }` into `skipped[]`, continue.
   - Otherwise → add to in-memory set, push parsed row into `toInsert[]`.
8. **Bulk insert** — `prisma.attendee.createMany({ data: toInsert })` inside a transaction. Defaults: `consentGiven: true`, `consentedAt: now()`, `profileCompleted: true`, `role: <mapped from Category>`.
9. **Audit log** — single `AuditLog` row: `action: 'ATTENDEES_BULK_IMPORTED'`, metadata `{ added, skipped: skipped.length, errors: errors.length, totalRows }`. Mirrors the existing `ATTENDEE_INVITED` audit pattern.
10. **Response** —

```json
{
  "added": 47,
  "skipped": [
    { "row": 12, "email": "priya@acme.com", "reason": "ALREADY_REGISTERED" },
    { "row": 18, "email": "rahul@xyz.io",   "reason": "DUPLICATE_IN_FILE" }
  ],
  "errors": [
    { "row": 5,  "reason": "INVALID_EMAIL", "value": "rahulwest@@gmail" },
    { "row": 31, "reason": "INVALID_CATEGORY", "value": "speaker" }
  ]
}
```

**Why synchronous:** A 1000-row .xlsx parses in under 1 second with `exceljs`, validates in milliseconds, and `createMany` inserts in one round-trip — comfortably within the existing 30s API timeout. No queue, no polling, no job-status table. Future YAGNI revisit if events ever need 10k+ rows.

**Why one transaction for the insert:** All-or-nothing semantics on the DB write. A mid-batch DB error doesn't leave the organizer with a half-imported list.

## 7. Error handling

### Layer 1 — Request-level (whole upload rejected)

| Trigger | HTTP | Code | Modal message |
|---|---|---|---|
| No file in body | 400 | `NO_FILE` | "No file selected. Choose an .xlsx file and try again." |
| Wrong mimetype / not `.xlsx` | 400 | `INVALID_FILE_TYPE` | "Only .xlsx files are accepted. Re-export from Excel/Sheets." |
| File > 5 MB | 413 | `FILE_TOO_LARGE` | "File is too large. Max 5 MB. Split your list and try again." |
| Worksheet `Attendees` missing | 400 | `INVALID_TEMPLATE` | "Sheet 'Attendees' not found. Re-download the template." |
| Headers don't match template | 400 | `INVALID_TEMPLATE` | Lists missing/extra columns by name. "Re-download template" CTA. |
| > 1000 data rows | 413 | `TOO_MANY_ROWS` | "Max 1000 rows per upload. Split your file and re-upload." |
| Organiser doesn't own event | 403 | (Nest default) | Generic "Not authorized". |

Response shape: `{ statusCode, code, message, details? }`. Modal stays open so the organizer can fix and retry without losing context.

### Layer 2 — Per-row failures (some rows succeed, results panel itemizes)

| Reason code | Meaning |
|---|---|
| `MISSING_REQUIRED_FIELD` | A required cell is empty. `details` names the column. |
| `INVALID_EMAIL` | Email doesn't match RFC pattern. |
| `INVALID_CATEGORY` | Category cell missing or not in the dropdown set. |
| `FIELD_TOO_LONG` | A string field exceeds its column max (e.g. firstName > 80 chars). |
| `INVALID_URL` | LinkedIn / Website / Photo URL doesn't parse as `http`/`https`. |
| `ALREADY_REGISTERED` | Email already exists in this event (skipped). |
| `DUPLICATE_IN_FILE` | Same email appears earlier in this upload (skipped). |

### Layer 3 — Mid-insert failures

Single `createMany` is wrapped in a Prisma transaction.

- Transaction rolls back → no partial state in DB.
- Server logs the error with row count + organiser ID.
- Response: `500 IMPORT_FAILED` — "Import failed, no attendees were added. Please try again."
- Audit log NOT written on failure.

### Photo URL safety

The Photo URL column is the one place untrusted input becomes a rendered `<img src="">`. Validation at import time rejects anything other than `http:` or `https:` URLs (blocks `javascript:`, `data:`, etc.). The directory's existing image rendering trusts `profilePhotoUrl` as-is — so the validation MUST happen at import, not display.

### Concurrency edge case

Two organizers of the same event upload overlapping spreadsheets simultaneously. Layer-2's pre-fetched email set could miss a row the other upload inserts moments later. The DB unique constraint `(eventId, email)` is the safety net — `createMany` will throw `P2002` on collision.

Recovery:
1. Catch the Prisma `P2002` error.
2. Re-query the existing emails for this event (a fresh `SELECT email FROM attendees WHERE eventId = $1`).
3. Filter `toInsert` against the new set — any newly-found colliders move to `skipped[]` with reason `ALREADY_REGISTERED`.
4. Retry `createMany` once with the filtered list, in a fresh transaction.
5. If it fails again with `P2002` (a third concurrent organizer kept inserting), surface as `IMPORT_FAILED`. We don't loop further — at that point something abnormal is happening and the organizer should retry manually.

Prisma's `P2002` doesn't include the colliding value, which is why we re-query rather than try to parse it from the error.

### Frontend network failures

If the upload XHR fails (network drop, 502 from a proxy), the modal shows: "Connection lost. Your file was not uploaded. Try again." The file stays selected — no auto-retry, since uploads are not idempotent and silent retry could double-import.

## 8. Visual design — the badge

A shared component: `apps/web/src/components/profile/RoleBadge.tsx`.

Renders nothing for `role: ATTENDEE` (the default — no badge clutter on regular attendees). For SPONSOR / KEY_SPEAKER / KEY_ORGANIZER it renders:

- A small filled star icon overlaid on the bottom-right of the avatar (corner-badge style — like Instagram's verified tick).
- A colored pill chip beside the name with the category text and a star icon prefix:
  - `SPONSOR` → amber chip (`bg-amber-100 text-amber-800`)
  - `KEY_SPEAKER` → indigo chip (`bg-indigo-100 text-indigo-800`)
  - `KEY_ORGANIZER` → purple chip (`bg-purple-100 text-purple-800`)

Accessibility: each badge has `aria-label="Sponsor"` / `"Key speaker"` / `"Key organizer"` so screen readers announce the role rather than treating it as decoration.

The component takes the role + the avatar element as children/props. It composes around existing avatar JSX rather than replacing it — the directory's existing `<img>` and initials-fallback patterns stay.

## 9. Testing strategy

### API unit tests — `apps/api/src/attendees/attendees.service.spec.ts`

- Header validation: correct → proceeds; missing column → `INVALID_TEMPLATE`; extra column → `INVALID_TEMPLATE`.
- Per-row validation: invalid email → `errors[]`; lowercase "speaker" Category → `INVALID_CATEGORY`; valid row → `toInsert[]` with `profileCompleted: true` and the right `role`; `Photo URL: javascript:alert(1)` → `INVALID_URL`.
- Duplicate handling: existing email → `skipped[]` with `ALREADY_REGISTERED`; same email twice in file → second occurrence → `skipped[]` with `DUPLICATE_IN_FILE`, first inserted.
- Row-count guard: 1001 data rows → `TOO_MANY_ROWS`.
- Audit log: success writes one `ATTENDEES_BULK_IMPORTED` row with the right metadata.
- Failure: simulated Prisma error → no audit log, response is `IMPORT_FAILED`.

### API e2e — `apps/api/test/attendees-import.e2e-spec.ts`

One happy-path test: generate template via `GET /import-template`, parse with `exceljs`, fill 5 rows (one per category + one duplicate), `POST` to `/import`, assert response shape, assert DB rows landed with the right `role`.

### Web component tests

`apps/web/src/components/profile/RoleBadge.test.tsx`:
- `ATTENDEE` → renders nothing.
- `SPONSOR` / `KEY_SPEAKER` / `KEY_ORGANIZER` → renders the right color chip + star.
- `aria-label` is set per role.

`apps/web/src/components/organiser/BulkImportAttendeesModal.test.tsx`:
- Step 1 → Step 2 transition.
- Client-side rejection of non-`.xlsx` files.
- Successful upload → results panel renders the counts.
- Mocked 413 → banner with the right message, modal stays on Step 2 with file still selected.

### Manual UI verification

(Per the project's CLAUDE-level guideline: test in a browser before reporting complete.)

1. Organiser dashboard → event → attendees page → "Add attendees" → download template, fill 3 rows (Sponsor, Speaker, Organizer), upload, see results panel, close.
2. Confirm new rows appear in the attendees table with the right Role chip.
3. Log in as an imported attendee → directory shows other imported attendees with the star + chip badge.
4. Open a Sponsor's public profile → badge renders.
5. Send + accept a connection between an Attendee and a Sponsor → connections list shows the badge.
6. Verify a regular self-registered attendee (no role) shows no badge — i.e. existing flows are visually unchanged.

### What is NOT tested

- Visual regression / pixel snapshots — overkill, project has none today.
- Load testing massive files — guarded by 5 MB / 1000 row caps.
- Playwright e2e — project has no Playwright setup; manual checklist covers the full path.

## 10. Out of scope

- Editing or deleting an attendee's role after import (could be a follow-up).
- Re-uploading the same Excel to update existing attendees — `ALREADY_REGISTERED` skip is intentional per Decision §2.
- Sending email invites to imported attendees — explicitly chosen against (silent import).
- Multi-role per person — explicitly chosen against (single category).
- A background-job pipeline for >1000 row imports — YAGNI; revisit if ever needed.
- Attendee-side ability to set their own role — roles are organizer-managed only.
